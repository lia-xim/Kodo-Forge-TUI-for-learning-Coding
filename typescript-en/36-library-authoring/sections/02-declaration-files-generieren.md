## Background: What are Declaration Files?

> **Origin Story: The .d.ts Format**
>
> When TypeScript was released in 2012, millions of JavaScript libraries
> already existed without any type information. Anders Hejlsberg and his
> team needed a way to add types to existing JavaScript code —
> WITHOUT changing the code. The solution: .d.ts files.
>
> .d.ts stands for "declaration TypeScript". These files contain ONLY
> type information — no executable code. They are the equivalent of
> C/C++ header files: they describe the interface, not the
> implementation. For library authors, .d.ts files are the most important
> artifact alongside the JavaScript code itself.

When you publish a TypeScript library, you deliver two things:

1. **JavaScript code** (.js) — gets executed
2. **Declaration files** (.d.ts) — get read by TypeScript

Your consumers run the .js code, but their IDE and compiler read the
.d.ts files for autocomplete, error checking, and documentation.

## Enabling declaration generation

```typescript annotated
// tsconfig.json for library authoring
{
  "compilerOptions": {
    "declaration": true,
    // ^ Generates .d.ts files alongside the .js files
    // ^ REQUIRED for libraries

    "declarationMap": true,
    // ^ Generates .d.ts.map files
    // ^ Enables "Go to Definition" → .ts source code instead of .d.ts
    // ^ RECOMMENDED — greatly improves DX for consumers

    "sourceMap": true,
    // ^ Generates .js.map files for runtime debugging

    "outDir": "./dist",
    // ^ All generated files end up here

    "rootDir": "./src",
    // ^ Source root — determines the directory structure in dist/

    "strict": true
    // ^ Libraries should ALWAYS use strict mode
  },
  "include": ["src/**/*"]
}
```

What gets produced:

```
  src/
    index.ts          ← Your source code
    utils.ts

  dist/               ← Generated files
    index.js          ← JavaScript (executable)
    index.d.ts        ← Types (for consumers)
    index.d.ts.map    ← Declaration map (Go to Definition)
    index.js.map      ← Source map (debugging)
    utils.js
    utils.d.ts
    utils.d.ts.map
    utils.js.map
```

> 🧠 **Explain it to yourself:** Why do you need BOTH .js.map and .d.ts.map? What does each file do?
> **Key points:** .js.map: mapping from dist/index.js → src/index.ts (for runtime debugging) | .d.ts.map: mapping from dist/index.d.ts → src/index.ts (for "Go to Definition" in the IDE) | Without .d.ts.map the IDE jumps to the .d.ts instead of the source code

## emitDeclarationOnly: When a bundler produces the JS code

If you use tsup, esbuild, or another bundler for your JavaScript
output, you only need tsc for the .d.ts files:

```typescript annotated
// tsconfig.json — generate only declarations
{
  "compilerOptions": {
    "declaration": true,
    "declarationMap": true,
    "emitDeclarationOnly": true,
    // ^ Generates ONLY .d.ts and .d.ts.map — NO .js!
    // ^ The bundler (tsup, esbuild) produces the .js output
    "outDir": "./dist"
  }
}

// Build script:
// 1. tsup src/index.ts --format cjs,esm  ← generate JavaScript
// 2. tsc --emitDeclarationOnly            ← generate .d.ts
// ^ Together: fast JS build + correct types
```

> 💭 **Think about it:** Why not just use tsc for everything?
> Why go through a bundler at all?
>
> **Answer:** tsc produces one .js file PER .ts file — it doesn't bundle.
> For libraries you often want a single bundle (or CJS + ESM).
> Bundlers can do: tree-shaking, minification, code-splitting. tsc cannot.
> So: bundler for JS, tsc for .d.ts.

## What belongs in the .d.ts — and what doesn't?

TypeScript generates .d.ts from your public API:

```typescript annotated
// src/index.ts — Your source code
export interface User {
  id: string;
  name: string;
  email: string;
}

export function createUser(name: string, email: string): User {
  // Implementation — does NOT appear in .d.ts
  const id = crypto.randomUUID();
  console.log(`Creating user ${name}`);
  return { id, name, email };
}

// Private helper function — does NOT appear in .d.ts
function validateEmail(email: string): boolean {
  return email.includes("@");
}

// dist/index.d.ts — Generated by tsc
// export interface User {
//   id: string;
//   name: string;
//   email: string;
// }
// export declare function createUser(name: string, email: string): User;
//
// ^ Only the SIGNATURE — no implementation
// ^ Private functions (not exported) are omitted entirely
// ^ Interface remains complete (has no implementation)
```

> ⚡ **Framework connection (Angular):** Angular libraries (created with
> `ng generate library`) use a special ng-packagr configuration
> that automatically generates correct .d.ts files and the `exports` field.
> When building an Angular library, ng-packagr handles everything
> shown manually in this lesson. But understanding what happens underneath
> helps with debugging.

## Quality control for .d.ts

Generated .d.ts files can have issues. Check these points:

```typescript annotated
// Problem 1: Internal types get exposed
// src/internal.ts
type InternalConfig = { secret: string };  // Not exported!

// src/index.ts
export function getConfig(): InternalConfig {
  // ^ WARNING: Return type references a non-exported type
  // ^ In .d.ts: TypeScript "inlines" the type or shows an error
  return { secret: "abc" };
}
// Fix: export InternalConfig OR specify the return type explicitly
export function getConfig(): { secret: string } {
  return { secret: "abc" };
}

// Problem 2: Types that are too broad
export function parseConfig(json: string) {
  return JSON.parse(json);
  // ^ Return type: any! That's bad for consumers
}
// Fix: Explicit return type
export function parseConfig(json: string): Config {
  return JSON.parse(json) as Config;
}

// Problem 3: Missing generic constraints
export function first<T>(items: T[]) {
  return items[0];
  // ^ Return type: T — OK, but could be T | undefined
}
// Fix: Honest return type
export function first<T>(items: T[]): T | undefined {
  return items[0];
}
```

> 🧪 **Experiment:** Generate .d.ts files and inspect them:
>
> ```bash
> # 1. Create src/lib.ts:
> # export function add(a: number, b: number) { return a + b; }
>
> # 2. Generate .d.ts:
> npx tsc src/lib.ts --declaration --emitDeclarationOnly --outDir dist
>
> # 3. Inspect the result:
> cat dist/lib.d.ts
> # → export declare function add(a: number, b: number): number;
>
> # 4. Change to: export function add(a, b) { return a + b; }
> # 5. Generate again — what happens to the return type?
> ```
>
> Observe: Without type annotations TypeScript infers. The result
> can be surprising — explicit types in libraries are safer.

## Declaration Maps: Go to Definition

Declaration Maps are the feature that makes the biggest DX difference
for your consumers:

```typescript annotated
// Without declarationMap:
// Consumer presses "Go to Definition" on createUser()
// → IDE jumps to: node_modules/my-lib/dist/index.d.ts
// → Shows: export declare function createUser(name: string, email: string): User;
// → No comments, no implementation, not helpful

// With declarationMap:
// Consumer presses "Go to Definition" on createUser()
// → IDE jumps to: node_modules/my-lib/src/index.ts (!)
// → Shows: the FULL source code with comments
// → Maximum transparency and documentation
```

For this to work, the source code must be included in the package:

```typescript annotated
// package.json — include source code in the npm package
{
  "files": [
    "dist",
    // ^ JavaScript + .d.ts + maps
    "src"
    // ^ Source code for declaration maps
    // ^ OPTIONAL: Only needed for "Go to Definition" into source
    // ^ Increases package size — weigh the trade-off
  ]
}
```

## What you've learned

- `declaration: true` generates **.d.ts files** from your source code
- `declarationMap: true` enables **"Go to Definition" into the source** instead of into the .d.ts
- `emitDeclarationOnly` is ideal when a **bundler produces the JS output**
- **.d.ts quality** check: no `any` return types, no exposed internals, explicit types
- **Include source code** (files: ["src"]) for maximum developer experience

**Core concept to remember:** .d.ts files are your library's business card. They determine how consumers experience your library — autocomplete, error messages, documentation. Invest in their quality: explicit types, declaration maps, and no `any` leaks.
